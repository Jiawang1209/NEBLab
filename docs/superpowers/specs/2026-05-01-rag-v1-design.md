# RAG 知识库 v1 设计文档

**项目**：NEBLab — 北方生态屏障数字实验室 RAG 知识库
**日期**：2026-05-01
**阶段**：v1 MVP
**状态**：待用户审阅

---

## 1. 项目背景

依托"森林生态与保育"重点实验室，围绕北方沙地生态治理（科尔沁、浑善达克歼灭战）建设数字实验室。RAG 知识库是数字实验室"知识发现"层的入口模块，目标是把领域文献转化为科研人员**可问、可引、可信**的知识库。

**重要边界**：本文档仅覆盖 RAG 知识库子系统（v1）。母项目还包括 5 个其他子系统（IoT 数据采集、多源异构数据库、可视化平台、多模态助手、决策模拟系统），均不在本文档范围内，建议按"先 RAG → 再数据库 → 再可视化"的顺序串行推进。

## 2. v1 范围

### 2.1 包含的功能（5 个核心用例）

| 编号 | 用例 | 描述 | 示例查询 |
|------|------|------|---------|
| ① | 文献综述 | 多文档合成式回答 | "目前科尔沁沙地植被恢复的主流观点有哪些？" |
| ② | 方法查找 | 检索专业方法/技术 | "用什么方法测沙地土壤含水率？" |
| ⑤ | 写作辅助 | 给写作场景找参考文献 | "我在写浑善达克引言，给我 5 篇关键引用" |
| ⑦ | 跨语种检索 | 中英互通问答 | 中文问，从英文文献检索作答（或反之） |
| ⑧ | 教学/培训 | 给研究生用，含术语解释 | "什么叫风沙流？给我 3 篇入门文献" |

**共享技术核心**：5 个用例共用同一套向量检索 + 重排 + LLM 生成内核，差异主要在 prompt 模板和 UI 层。

### 2.2 明确不包含（推迟到 v2/v3）

- ③ 数据/参数提取（需 PDF 表格抽取，工程量大）
- ④ 趋势分析（属文献计量学范畴，应做成独立工具）
- ⑥ 假设核验（需立场分析）
- 实验室内部观测数据（v2 加入，需重新评估部署形态）
- 多模态：图表/语音/虚拟助手（v2/v3）
- 知识图谱集成（v2 评估）
- 移动端适配、多用户权限

## 3. 语料采集计划

### 3.1 目标体量（v1 = 方案 B 平衡版）

- **摘要级索引**：5000 篇（OpenAlex 元数据 + 摘要全收）—— 用于覆盖广度
- **全文级索引**：2500 篇（从 5000 候选中按主题优先级精选）—— 用于深度问答
- **学位论文**：~250 篇（中文为主，985/211 + 中科院 + 优博）
- **灰文献**：~25 篇（核心权威报告）

**全文文档总计 ≈ 2775 篇 + 5000 篇摘要**

> 设计原则：摘要级索引让系统**广度足够**（综述、找文献能跑），全文级索引让系统**深度可信**（精确引用、方法细节）。后续按需扩充全文，不停机。

### 3.2 主题配额（v1 全文级 = 2500 篇，摘要级 = 5000 篇）

**全文级配额**（按 50% 缩减原 5000 篇配额）：

| 主题 | 全文篇数 | 中文 | 英文 | 优先级 |
|------|---------|------|------|--------|
| 荒漠化监测 / 机理（含防沙治沙总论） | 550 | 300 | 250 | 🔴 必须 |
| 三北防护林 / 农田防护林 | 500 | 350 | 150 | 🔴 必须 |
| 科尔沁 / 浑善达克沙地具体研究 | 450 | 380 | 70 | 🔴 必须 |
| 无人机 / LiDAR 植被遥感 | 300 | 100 | 200 | 🟡 想要 |
| 林草生态系统结构与功能 | 250 | 150 | 100 | 🟡 想要 |
| 水土保持 / 草地退化 / 干旱区生态 | 250 | 150 | 100 | 🟡 想要 |
| 山水林田湖草沙系统耦合 | 200 | 180 | 20 | 🟡 想要 |

### 3.3 质量过滤规则

- **时间范围**：不限年份（用户选择 a1）
- **英文期刊**：JCR Q1/Q2 + 行业旗舰；**2000 年前**改用"核心期刊 + 经典文献白名单"补丁（JCR 在 1997 年才有）
- **中文期刊**：CSSCI / CSCD / 北大核心；2000 年前同样使用白名单补丁
- **排除**：会议短摘、新闻稿、综述类工具书、低质灰文献

### 3.4 来源策略（三层架构）

```
┌─────────────────────────────────────────────────────────┐
│ 层 1：元数据 100% 自动（无 API key）                     │
│        OpenAlex + Crossref + Unpaywall                  │
│        → 几小时建 ~8000 候选池，去重后 ~5000（全部入摘要库）│
│                                                          │
│ 层 2：英文全文 半自动（需用户申请 API key）              │
│        Elsevier TDM + Wiley TDM + Springer + Unpaywall  │
│        → 目标 890 篇英文全文                              │
│                                                          │
│ 层 3：中文全文 半自动（用户登录 + 浏览器自动化）         │
│        用户 CNKI/万方/维普 网页批量导出题录 RIS          │
│        → 浏览器自动化脚本用 session 限速下 PDF（<200/日）│
│        → 目标 1610 篇中文全文（10 天达标）                │
└─────────────────────────────────────────────────────────┘
```

### 3.5 采集时间线（6 周，与开发并行）

| 周 | 用户负责 | 系统负责 |
|----|---------|---------|
| W1 | 申请 Elsevier API key（优先）；测 CNKI 登录 | 制订 7 主题关键词词表；OpenAlex/Crossref 拉 5000 候选元数据 + 去重 → **摘要级 RAG 当周即可跑通** |
| W2 | CNKI 批量导出 ~1500 题录；启动半自动下载 | Elsevier TDM 启动拉取（目标 500 英文）；MinerU + deepseek-ocr 解析 pipeline 上线 |
| W3 | 周期性 CNKI 网页导出，监控 PDF 下载 | 中文 PDF 持续入库（~800/周）；英文 TDM 累计到 700 |
| W4 | 周期性 CNKI 网页导出（最后冲刺） | 中文累计 1610 + 英文 890 = **2500 全文入库完毕**；学位论文 250 + 灰文献 25 |
| W5 | 抽查质量 | 跨语种去重、质量清洗、最终筛选 |
| W6 | 用户验收 | 全量入库稳定 + RAG MVP 端到端跑通 |

## 4. 技术栈

### 4.1 LLM 层（多模型路由）

| 角色 | 模型 | 用途 |
|------|------|------|
| 主力对话 | `deepseek-v3.2` / `deepseek-v4-flash` | 95% 流量，覆盖 ②⑤⑦⑧ |
| 深度推理 + 长综述 | `deepseek-r1:671b-64k` | ① 文献综述；用户标记"深度问题" |
| 轻量任务 | `deepseek-r1:32b` / `deepseek-v4-flash` | 查询改写、术语解释、教学注解 |
| 故障兜底（接口预留） | `qwen3:235b` / `spark-x1:70b` / `minimax-m27` | 单点故障 30 分钟切换 |

### 4.2 检索层

| 模块 | 主用 | 备选 |
|------|------|------|
| Embedding | `qwen3-embedding:8b` | `bge-large-zh` / BGE-M3 |
| Reranker | `qwen3-reranker:8b` | `bge-reranker-v2-m3` |
| 检索策略 | 混合（dense + sparse）→ rerank → top-k | — |

### 4.3 文档解析层

| 类型 | 主用 | 兜底 |
|------|------|------|
| 中文 PDF（学术排版） | MinerU | `deepseek-ocr` |
| 英文 PDF | LlamaParse | `deepseek-ocr` |
| 扫描件 / 图像化 PDF | `deepseek-ocr` | — |

### 4.4 数据层

| 用途 | 选择 |
|------|------|
| 向量库 | Qdrant Cloud（备选阿里云向量检索） |
| 元数据 + 全文存储 | PostgreSQL 16（云托管） |
| 文件存储（PDF） | 阿里云 OSS |

### 4.5 应用层

| 层 | 选择 |
|-----|------|
| RAG 框架 | LlamaIndex 核心 + 关键路径自写（避免被框架锁死） |
| 后端 API | FastAPI（Python） + Server-Sent Events 流式 |
| 前端 | Next.js 15 + shadcn/ui + Tailwind 4 |
| 可观测性 | Langfuse（开源 RAG observability） |
| 评测框架 | RAGAS + 自写 pytest 评测套件 |

### 4.6 多模型 Provider 抽象（关键架构原则）

```
LLMProvider (interface)
├─ DeepSeekProvider     ← v1 主用
├─ QwenProvider         ← 接口预留
├─ SparkProvider        ← 接口预留
└─ MiniMaxProvider      ← 接口预留

EmbeddingProvider (interface)
├─ Qwen3EmbProvider     ← v1 主用
├─ BGEProvider          ← 备选
└─ ...

RerankerProvider (interface)
├─ Qwen3RerankerProvider ← v1 主用
├─ BGERerankerProvider   ← 备选
└─ ...

ParserProvider (interface)
├─ MinerUProvider
├─ LlamaParseProvider
└─ DeepSeekOCRProvider
```

业务逻辑只依赖接口，不绑定具体模型/产品。后续切换、A/B 测试、新模型上线都不需要改业务代码。

## 5. 端到端数据流

### 5.1 入库 pipeline（离线）

```
PDF 文件
   ↓
[ParserProvider]
   ├─ MinerU（中文）
   ├─ LlamaParse（英文）
   └─ deepseek-ocr（兜底）
   ↓
结构化文本（章节 / 段落 / 表格元数据）
   ↓
[多策略 chunking]（用户计划书第 4 节(2)5 种方法）
   ├─ 固定（512 token）
   ├─ 语义（按段落语义边界）
   ├─ 递归（结构化层级）
   ├─ 文档结构（按章节）
   └─ 智能（LLM-aided）
   ↓
[EmbeddingProvider] qwen3-embedding:8b
   ↓
向量入 Qdrant；元数据入 PostgreSQL；PDF 存 OSS
```

### 5.2 查询 pipeline（在线）

```
用户问题
   ↓
[查询改写] LLM 扩展同义词、双语翻译（⑦ 跨语种）
   ↓
[混合检索] dense + sparse → top-50
   ↓
[RerankerProvider] qwen3-reranker:8b → top-5/10
   ↓
[场景路由]
   ├─ ① 综述：deepseek-r1:671b-64k（长上下文）
   ├─ ②⑤⑦⑧：deepseek-v3.2 / v4-flash
   └─ "深度模式"按钮 → r1 系列
   ↓
[LLM 生成] 流式输出，强制 inline 引用 [1][2]
   ↓
[引用校验] 自动检查 [1][2] 是否真出现在 retrieval 结果
   ↓
SSE 流式返回前端 → 字符级渲染
```

## 6. UI 设计

### 6.1 双形态实现（A + B）

```
默认：B 形态（混合式）
┌────────────┬──────────────┬────────────┐
│ 聊天对话区 │ 引用文献卡片 │ 文献库浏览 │
│ (流式回答) │ (PDF 预览)   │ (主题分类) │
└────────────┴──────────────┴────────────┘

[专注模式] 按钮  →

A 形态（纯对话）
┌────────────────────────────────────────┐
│ 聊天对话区（侧栏 CSS 隐藏）            │
└────────────────────────────────────────┘
```

**实现策略**：单一 React 组件树，CSS 控制侧栏折叠。同一套代码、两种布局，开发成本接近 B 单独。

### 6.2 关键交互

- **流式回答**：SSE 字符级输出
- **引用卡片**：[1] 可点击 → 弹出原文 chunk + PDF 跳转链接
- **深度模式按钮**：路由到 `deepseek-r1:671b-64k`
- **跨语种自动**：用户中文问 → 英文文献也召回 → 回答用中文 + 英文文献条目（标注 [EN]）
- **教学模式**（⑧）：检测专业术语 → tooltip 自动解释 + "推荐入门文献" 链接
- **写作辅助模式**（⑤）：用户粘贴写作上下文 → 系统给"建议引用此处" + 引用候选

## 7. 评测方案

### 7.1 评测集规模与构成（v1 = 100 题平衡版）

- **总计 100 题**，覆盖 5 个用例（每用例 20 题）
- **难度分级**：简单 30% / 中等 50% / 困难 20%
- **来源（三结合）**：
  - **AI 自动生成 50 题**：基于语料元数据 + LLM 出题（用 deepseek-r1）
  - **用户 + 研究生审核扩充 25 题**：在自动题基础上修订 + 补充
  - **关键场景手写 25 题**：典型咨询 / 答辩高频问题 / 边界 case

> v2 扩充至 200 题（按上一版方案），用于发表"系统评估"章节。

### 7.2 核心质量指标（用户选择 i）

| 指标 | 目标 | 衡量方法 | 优先级 |
|------|------|---------|--------|
| **引用准确率** | **≥ 95%（硬门槛）** | LLM-as-judge 自动校验 [1][2] 是否真支持论断 + 5% 人工抽检 | 🔴 必达 |
| 检索相关性 | ≥ 85% | RAGAS `context_relevance` + Recall@10 | 🟡 软指标 |
| 生成质量 | 平均 ≥ 4.0 / 5 | 5 分制人工评分（信息完整 / 表达专业 / 无幻觉） | 🟡 软指标 |
| 回答延迟（首 token） | P95 < 2s | Langfuse 追踪 | 🟡 软指标 |
| 完整回答延迟 | P95 < 8s | Langfuse 追踪 | 🟡 软指标 |

### 7.3 评测自动化

- **25 题 smoke set**：每次 PR 自动跑（CI 集成）
- **100 题 full set**：每次发布前跑
- **周报机制**：选 10-15 题人工抽检 → 发现问题 → 评测集迭代

## 8. 整体实施时间线（6 周平衡版）

| 阶段 | 周 | 主要产出 |
|------|----|---------|
| Sprint 0：基建 + 摘要 RAG | W1 | 仓库结构、CI、Provider 抽象层、采集脚手架；OpenAlex 5000 元数据入库；**摘要级 RAG 当周可演示** |

## 9. 风险与权衡

| 风险 | 等级 | 缓解 |
|------|------|------|
| CNKI 反爬触发，影响学校访问权 | 🔴 高 | 严格限速（< 200 PDF/天）；用浏览器 session 而非 API；分主题分批；中断即停 |
| DeepSeek API 单点故障 | 🟡 中 | Provider 抽象层；30 分钟切到 Qwen / Spark / MiniMax |
| 中文 PDF 解析质量不稳 | 🟡 中 | MinerU + deepseek-ocr 双管线；关键论文人工校核 |
| 评测集开始太晚 → 后期返工 | 🟡 中 | W6 就启动第一版评测，与开发并行 |
| 5000 篇语料周期超出预期 | 🟢 低 | 系统在 500 篇时就能跑，不阻塞开发 |
| v2 加内部数据需要重做架构 | 🟢 低 | Provider 抽象 + 数据源 adapter 已预留接口 |

## 11. 决策摘要（5 句话）

1. **范围**：v1 = 5 个用例（综述、方法、写作、跨语种、教学），其他推迟。
2. **语料**：5000 篇摘要 + 2500 篇全文 + 250 学位 + 25 灰文献，**6 周分三层采集**（方案 B 平衡版）。
3. **技术栈**：DeepSeek 系列 LLM + Qwen3 emb/reranker + Qdrant + LlamaIndex + Next.js，全云 API，多模型 Provider 抽象。
4. **UI**：默认混合式（聊天 + 文献库 + 引用卡片）；可切纯对话专注模式。
5. **质量**：100 题评测集，引用准确率 ≥ 95% 是硬门槛。
